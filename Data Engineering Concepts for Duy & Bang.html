1. Data Storage: Lake vs. Mart

Data Lake

A **Data Lake** is a centralized repository that stores **all** your data—structured, semi-structured, and unstructured—at any scale. It's often used for **storing raw data** "as is," without first structuring it.

  • **Data Schema:** Schema-on-Read (define structure when reading).
  • **Primary Users:** Data Scientists, Machine Learning Engineers.
  • **Purpose:** Big Data analytics, discovery, and advanced modeling.

Data Mart

A **Data Mart** is a simpler, highly focused subset of a Data Warehouse (or Data Lake). It is designed to serve a **specific business function** or team, such as Sales, Finance, or Marketing.

  • **Data Schema:** Schema-on-Write (highly structured when written).
  • **Primary Users:** Business Analysts, Executive Reporting.
  • **Purpose:** Simple, fast reporting and dashboarding for specific departments.

2. Data Flow: ETL and ELT

ETL (Extract, Transform, Load)

This is the traditional method:

  1. **Extract:** Grab data from source systems (databases, APIs).
  2. **Transform:** Clean, aggregate, and enrich the data **before** it moves to the destination.
  3. **Load:** Write the transformed data to the target system (e.g., a Data Mart).
**Key Point:** Transformation happens outside the final database.

ELT (Extract, Load, Transform)

The modern approach, favored with powerful cloud data warehouses:

  1. **Extract:** Grab data from source systems.
  2. **Load:** Load the raw data directly into the target (e.g., a Data Lake or cloud warehouse).
  3. **Transform:** Clean and model the data **using the power of the target system**.
**Key Point:** Transformation happens inside the final database.

3. Data Pipelines

A **Data Pipeline** is the automated set of processes that moves data from one system to another. It's the engine that powers both ETL and ELT jobs.

  • **Automation:** Runs on a schedule or triggers based on events (e.g., a new file arriving).
  • **Orchestration:** Tools like Apache Airflow manage the dependencies and sequence of tasks in the pipeline.
  • **Reliability:** Pipelines must include logging, monitoring, and error handling to ensure data integrity.

4. NoSQL Databases: Cassandra and HBase

Apache Cassandra

**Cassandra** is a distributed **NoSQL database** designed to handle massive amounts of data across many commodity servers, providing high availability with no single point of failure. It is often described as a **wide-column store**.

  • **Key Feature:** **High Availability** and **Linear Scalability**. You can add more nodes (servers) to increase capacity and performance.
  • **Use Case:** Ideal for time-series data, operational data with high write throughput (e.g., IoT, monitoring).
  • **Trade-off:** Uses **eventual consistency** to achieve speed and availability.

Apache HBase

**HBase** is a distributed, versioned, **non-relational database** modeled after Google's Bigtable. It runs on top of the **Hadoop Distributed File System (HDFS)**. It is also a **wide-column store**.

  • **Key Feature:** **Strict Consistency** (compared to Cassandra) and seamless integration with the Hadoop ecosystem.
  • **Use Case:** High-volume read/write access for large datasets, often integrated into complex Big Data platforms.
  • **Trade-off:** Generally more complex to set up and manage due to its dependency on HDFS.
